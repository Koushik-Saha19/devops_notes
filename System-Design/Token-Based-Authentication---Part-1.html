What is a Request Header?

Imagine you‚Äôre sending a physical letter. The request header is like the envelope of your letter. It contains crucial metadata and instructions about the letter itself, its sender, its recipient, and how it should be handled, before you even look at the actual content (the body of the letter).

In the context of web communication (specifically HTTP/HTTPS):

  • Metadata: Request headers are key-value pairs that carry supplementary information about the request being sent from a client (like your web browser or an application) to a server.
  • Purpose: They provide context and instructions necessary for the server to understand and process the request correctly.

Basically a header is metadata sent along with the request. It doesn‚Äôt contain actual data (like your login form info) ‚Äî instead, it provides extra context to the server about:

  • What the request is
  • How to process it
  • Who‚Äôs sending it
  • What format to expect
  • Whether the user is authenticated, etc.

üî∑ Why Token?

In a typical application, authentication means confirming who the user is. Traditionally, this happens using username and password. But:

  • You don‚Äôt want to send credentials with every request ‚Äî it‚Äôs insecure.
  • You want a way to verify identity for each API request without logging in again.
  • You want something the client can hold and present on every request, and the server can trust and verify.

üëâ This ‚Äúsomething‚Äù is called a token ‚Äî a short-lived, verifiable proof of identity.

üî∑ How Token Enables Authentication

  1. The user logs in once (e.g., with username/password).
  2. The authentication server verifies credentials.
  3. If valid, it generates a token and gives it to the user/client.
  4. From now on:
    • Every request includes the token in the header (Authorization: Bearer <token>).
    • The server checks if the token is valid and not expired.
    • If valid ‚Üí request is allowed.

üîÅ No need to re-login. The token replaces credentials for the duration of its lifetime.

üî∑ What is JWT (JSON Web Token)?

JWT is one type of token ‚Äî a standard format that is:

  • Self-contained (has all necessary user info inside it),
  • Digitally signed (so it can‚Äôt be tampered with),
  • And often used in stateless authentication systems.

‚úÖ Structure:

JWT is just a Base64-encoded string, made of 3 parts:

Header.Payload.Signature

Header:

  • Type (JWT)
  • Signing algorithm (e.g., HS256)
{
  "alg": "HS256",
  "typ": "JWT"
}

Payload:

  • Contains the actual claims like:
    • sub (subject, e.g., user ID)
    • name
    • roles
    • exp (expiration timestamp)
{
  "sub": "123456",
  "name": "John Doe",
  "roles": ["admin"],
  "exp": 1720000000
}

Signature:

  • Created by hashing the header + payload with a secret or private key.
  • This ensures data integrity ‚Äî if even 1 byte is changed, the signature won‚Äôt match.

üîí How JWT Enables Authentication

When a client sends a JWT with a request:

  1. The server:
    • Extracts the JWT from the request header.
    • Decodes the JWT.
    • Validates the signature using the shared secret or public key.
    • Checks if the token is expired (exp claim).
  2. If all checks pass ‚Üí request is authenticated.

No need to query the database every time ‚Äî it trusts the signed token.
